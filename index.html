Overview

Werunos provides full ext4 read-write support in userspace. It parses on-disk structures, replays the jbd2 journal when needed, and exposes a POSIX-like interface through WinFsp. The implementation emphasizes data integrity and crash safety for writes to raw disks and disk images.

Every ext4 filesystem is composed of block groups. Each group contains a block bitmap, an inode bitmap, an inode table, and data blocks. Werunos navigates this structure directly, reading and writing the superblock, group descriptors, inodes, extent trees, and directory entries at the block level.

◆ jbd2 journal replay, crash-safe redo log, and a 5-phase fsck are built into the driver itself.

Usage

enumeration
werunos devices

List all physical disks and their partition tables.

mount
werunos mount <letter> <disk> <part>

Mount an ext4 partition as a Windows drive letter. Blocks until Ctrl+C.

example werunos mount G: 0 4
integrity
werunos fsck [--fix] <device> [<part>]

5-phase integrity check across block bitmaps, inode bitmaps, inode structures, superblock counts, and directory entries. Pass --fix to auto-correct mismatched counts.

example werunos fsck --fix \\.\PhysicalDrive0 4
setup
werunos install

Download and install WinFsp. Requires Administrator privileges.

inspection
werunos <device> [<part>]

Without a subcommand, probe partitions on a device or list the root directory of an ext4 volume.

example werunos testfs.img
◆ Raw disk paths on Windows take the form \\.\PhysicalDriveN. Werunos replays the jbd2 journal and clears the recovery flag automatically on first mount after an unclean shutdown.

Known Boundaries

Limitation Impact Rationale
Depth-0 extents Files with >4 non-contiguous extents cannot extend Full B-tree manipulation is ~500 additional lines
No journal submission Metadata writes bypass jbd2; redo log covers crashes jbd2 transaction pipeline requires descriptor/data/commit
No EA writes Extended attributes preserved but not modified EA area lives beyond 128-byte base inode
No inline data EXT4_INLINE_DATA_FL inodes skipped on write Separate read/write paths needed
No encryption EXT4_ENCRYPT_FL files skipped Requires kernel keyring integration
No metadata checksums metadata_csum not verified or updated Checksum verification would add ~200 lines
No orphan_file Uses legacy S_last_orphan list Linux 5.15+ orphan file is separate linked-list
depth-0 The extent tree is stored inline in the inode (i_block, 60 bytes). A 12-byte header plus four 12-byte extents fit. Index blocks for deeper trees are not yet implemented.
redo log On every WriteAt: pre-read the target block, write old data to a log file in %TEMP%, fsync, issue the write, then truncate the log. On restart after a crash, the log is replayed to restore pre-write state.
testing Prefer disk images over real drives. ext4 is complex and there are no absolute guarantees. The redo log provides crash recovery, but always back up important data.
