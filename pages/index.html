Overview

Werunos provides high-performance ext4 and btrfs read-write support in userspace. It mounts volumes directly as native Windows drives via WinFsp, utilizing structured B-tree walks, logical-to-physical chunk translation, and transparent compression segment decoding without any kernel modules.

While maintaining lightweight ext4 mounts, Werunos features a complete userspace Btrfs engine. It decodes the superblock system chunk array, resolves complex chunk trees, performs in-place leaf mutations (insert, update, and delete), and supports transparent on-the-fly decompressions for zlib and LZO extents.

◆ Btrfs support includes custom leaf restructuring algorithms, CRC32c checksum recalculation, symbolic and hard link creation, and direct raw disk image (`.img`) mounts.

Capabilities

Ext4 & Btrfs Feature Comparison Matrix

Operation Ext4 Support Btrfs Support Technical Implementation / Detail
File Reading ✔ Yes ✔ Yes Ext4: extent B-tree walk, block caching. Btrfs: inline/regular extents, chunk tree logical maps.
File Writing ✔ Yes ✔ Yes Ext4: full block allocator, extent appending. Btrfs: inline/regular extents, dynamic block allocator with single-level B-tree leaf splitting.
Directory Ops ✔ Yes ✔ Yes Creating directories (with ./..), readdir traversal, directory entry unlinking.
File Creation / Deletion ✔ Yes ✔ Yes Ext4: inode allocation, group block reclaim. Btrfs: inode generation, leaf deletion of index/item keys.
Chmod / Chown ✔ Yes ✔ Yes Mutates permission mode, uid, and gid bits, recalculating CRCs/checksums dynamically.
File Truncation ✔ Yes ✔ Yes Supports resizing files (grow / shrink). Ext4: extent tree splitting. Btrfs: in-place leaf updates.
Links (Hard & Sym) ✔ Yes ✔ Yes Symlinks (fast/slow pathways). Hard links update link counts and references.
Filesystem FSCK ✔ Yes ✔ Yes Ext4: 5-phase validation and auto-repairs. Btrfs: superblock, chunk map, and recursive node CRC32c verifications.
Compression Support ✘ No ✔ Yes Btrfs: transparent decompression of zlib, LZO, and ZSTD extents, with dynamic Zlib write compression.

Usage

enumeration
werunos devices

List all physical disks and their partition tables.

mount
werunos mount <letter> <disk> <part>

Mount an ext4 or btrfs physical partition as a Windows drive letter. Blocks until Ctrl+C.

example werunos mount G: 0 4
mount image
werunos mount <letter> <imagePath>

Mount a raw ext4 or btrfs disk image file directly as a Windows drive letter.

example werunos mount Z: btrfs_test.img
integrity
werunos fsck [--fix] <device> [<part>]

5-phase integrity check across block bitmaps, inode bitmaps, inode structures, superblock counts, and directory entries (ext4 only). Pass --fix to auto-correct mismatched counts.

example werunos fsck --fix \\.\PhysicalDrive0 4
setup
werunos install

Download and install WinFsp. Requires Administrator privileges.

inspection
werunos <device> [<part>]

Without a subcommand, probe partitions on a device or list the root directory of an ext4/btrfs volume.

example werunos testfs.img
◆ Raw disk paths on Windows take the form \\.\PhysicalDriveN. Werunos automatically auto-detects filesystem types at runtime.

Known Boundaries

Limitation Impact Rationale
No journal submission Metadata writes bypass jbd2; redo log covers crashes (ext4) jbd2 transaction pipeline requires descriptor/data/commit
No btrfs CoW Leaf node writes updated in-place; bypasses transaction commits Crash-unsafe but lightweight, keeping tree structures simple (mitigated by SafeDevice redo logging)
No encryption EXT4_ENCRYPT_FL / encrypted files skipped Requires kernel keyring integration
No metadata checksums metadata_csum not verified or updated Checksum verification would add ~200 lines
btrfs writes Werunos fully supports both inline (up to 2KB) and regular extents (arbitrary sizes >2KB) on Btrfs, using physical chunk allocation and single-level B-tree leaf node splitting. Dynamic zlib write compression is automatically applied to compressible extents.
redo log On every Ext4 and Btrfs WriteAt: pre-read the target block (accelerated by a thread-safe 64-slot LRU block cache), write old data to a log file in %TEMP%, fsync, issue the write, then truncate the log. On restart after a crash, the log is replayed.
testing Prefer disk images over real drives. The redo log provides unified crash recovery, and the built-in fsck verifier can systematically validate Btrfs and Ext4 volume integrity. Always back up important data.
